Chapter one · Foundations

About this handbook

A reference guide for using simple-mcp as the shared hub for agentic development. Explains the conceptual model, the recommended patterns around it, and the 2026 Claude Code and Codex features that make the whole thing work in production.

simple-mcp is a local Model Context Protocol (MCP) server with a React 19 admin panel. It provides MCP tools for Jira and GitHub integrations, seven pre-built autonomous agents (Jira Triage, PR Review, Code Search, Sprint Planning, Local Repo Analysis, Confluence Reader, Database Explorer), an execution engine with guardrails, and both .claude/ and .codex/ configurations so Claude Code and Codex can share the same credentials, the same run history, and the same tool surface.

This handbook covers what belongs around simple-mcp to make that architecture hum: the hooks that turn repetitive guidance into deterministic guarantees, the skills that encode recurring recipes, the subagents that parallelize and isolate work, and the 2026 Claude Code features (routines, /simplify, /batch, Auto Mode) that changed what's worth building by hand.

Who this is for

Anyone using or contributing to simple-mcp. Three audiences specifically:

  • New users who've cloned the repo and want a mental model before setting up integrations
  • Daily users who want to go beyond agent_execute and build their own hook/skill/subagent stack around simple-mcp
  • Contributors who want context on where simple-mcp fits in the agentic ecosystem and what extensions would matter next
How to read this Parts I–II are sequential and foundational. Part III adds 2026 platform features on top. Part IV is daily operation. Part V is a reference for simple-mcp itself — what it provides today and proposed extensions. The appendices are quick reference.

The repo's Quick Start in the README covers actual setup (clone, install, configure integrations). This handbook covers what to build on top.

Three principles that run through every chapter

§ 01

Orchestrator and workers, not one big brain

A coding agent (Claude Code or Codex) stays the orchestrator. Subagents, second-model sessions, and simple-mcp's pre-built agents are workers. Workers have narrow scopes, their own context windows, and limited tool access. This is the only model that scales past one concurrent task.

§ 02

Context is the scarce resource — not tokens

Token optimization is downstream of context discipline. A clean main-session context beats every other optimization. Subagents exist primarily to absorb noisy work (file reads, test output, exploration) so the main reasoning stays focused.

§ 03

simple-mcp can be a database or a brain — choose deliberately

simple-mcp supports both modes. Server-side autonomous runs via agent_execute (the engine reasons with Anthropic, tracks progress). Client-driven tracked runs via agent_start_run / agent_record_step / agent_complete_run (Claude Code or Codex owns the reasoning loop, simple-mcp persists execution history). Use agent_execute for one-shot autonomy; use client-driven runs when the coding agent should stay in charge and you want unified observability without paying for two reasoning loops.

Every correction becomes a rule. Week one you correct ten things. Month three you correct zero. — the compounding loop

That's the payoff of investing in the layers around simple-mcp. An afternoon of setup, then every hour of work afterwards is cheaper and cleaner than the last, because your rules accumulate in hooks and skills instead of evaporating when a session ends.

Chapter two · Foundations

The mental model

Five primitives. Pick the right one for each job, and the whole system stays legible. Pick the wrong one and you're fighting the framework.

Before you touch a single config file, internalize what each primitive is for. The biggest source of wasted tokens and broken workflows in agentic development is using the wrong primitive for the job — encoding deterministic rules as prompts, encoding reasoning as hooks, encoding integration as skills.

The five primitives

PrimitiveWhat it isWhen to reach for itCost
Skill Directory with SKILL.md plus scripts, references, examples. Claude loads it on demand when the description matches. Reusable recipes you invoke by name or have Claude auto-trigger: "open a PR", "run tests", "turn a Jira ticket into a spec". Free until triggered
Subagent Markdown file in .claude/agents/ with YAML frontmatter. Own context window, own system prompt, own tool whitelist. Noisy or focused work you want to keep out of main context — exploration, code review, test runs, parallel implementation. Own model cost + input
Hook Shell command on a lifecycle event (PreToolUse, PostToolUse, Stop, etc.). Reads JSON via stdin, returns JSON via stdout. Deterministic rules that run every time: formatting, secret scanning, blocking destructive ops, recording to simple-mcp. Zero LLM tokens
Script Plain shell or Python file called from a skill, hook, or subagent. Implementation of anything that isn't reasoning. Parsers, formatters, merge logic, API calls. Zero
MCP tool Function exposed by an MCP server. For simple-mcp users, this is any tool from the simple-mcp server — Jira search, GitHub PR diff, agent execution. Integration with external systems: GitHub, Jira, Confluence, database, filesystem. Schema cost in input

If it's deterministic, make it a hook or script. If it needs judgment, make it a skill or subagent. If it needs external state, make it an MCP tool.

Why picking the right layer matters

Every primitive has a natural cost curve:

  • Hooks are free but rigid. They run every time and never hallucinate. But they can't reason — a hook can check that SQL starts with SELECT, but it can't judge whether a query is semantically safe.
  • Subagents are flexible but expensive. They reason well and isolate context, but each one costs tokens in their own window. Over-use them and your per-session cost explodes.
  • Skills are cheap until used. A 200-line SKILL.md doesn't cost anything until Claude decides to load it. This is why skills beat bloated CLAUDE.md files.
  • MCP tools are the most expensive per-turn. Their schemas live in every turn's input. Expose too many, you degrade every response.

The art is matching cost to value. Hooks for things that must happen 100% of the time. Skills for anything you do more than twice. Subagents for isolation. MCP tools for stateful integration. Scripts for everything else.

The shortcut When in doubt: ask yourself "can I write a shell script that does this?" If yes, you probably want a hook or a script. If no, you probably want a skill or subagent.
Chapter three · Foundations

The shared architecture

Four layers, one shared state. How Claude Code and Codex both plug into simple-mcp without duplicating anything.

simple-mcp resolves the hardest architectural problem in setups that want to run multiple coding agents: a single shared state layer that both Claude Code and Codex can write into. When Codex performs a tracked run via agent_start_run, Claude Code can see it later via agent_status. Same credentials, same run history, same tool surface. Zero duplication.

Everything in this handbook is built around preserving that property. The four-layer architecture:

What each layer owns

Layer 1 — Human interface

The user. In the terminal via Claude Code or Codex CLI, in Claude Desktop, on mobile via the Claude app, or in simple-mcp's React admin panel at localhost:3100. Goals get issued in natural language; plans get approved. Everything below this layer is in service of turning those words into verified, committed code.

Layer 2 — Reasoning agents

Claude Code and Codex can run in parallel. Claude Code reads CLAUDE.md and .claude/; Codex reads AGENTS.md and .codex/. The universal AGENTS.md is the single source of truth — CLAUDE.md references it and adds Claude-specific extras. Both are already present in the simple-mcp repo as working examples.

Layer 3 — The shared hub

simple-mcp itself. Not a third reasoning agent — a shared substrate. It records what the reasoning agents did (runs, steps, tasks, tokens), holds the encrypted credentials for integrations they use, and exposes a pre-built autonomous execution engine for workflows that should run without ongoing supervision. The React admin panel at localhost:3100 is the unified view.

Layer 4 — External services

GitHub, Jira, Confluence, databases, local filesystems. Credentials for each are stored once in simple-mcp, AES-256-CBC encrypted at rest. Neither Claude Code nor Codex ever handles raw tokens — they call MCP tools that call the services.

Architectural advantage A second user on the same machine running Codex sees the same runs in the admin panel as the first user running Claude Code. When someone asks "what did the deployment agent do yesterday?", there's one place to look.
Chapter four · Build the system

Bootstrap & layout

The files, folders, and environment variables. Set once, used forever. If you do nothing else in this handbook, do this chapter.

The project layout

Every project that wants to use simple-mcp gets this structure. The repo itself already follows it — treat heshansw/simple-mcp as a working reference and copy the pattern to downstream projects:

your-project/ ├── AGENTS.md # universal — Codex, Cursor, etc. ├── CLAUDE.md # thin wrapper → @AGENTS.md + extras ├── .mcp.json # points at simple-mcp ├── .claude/ │ ├── settings.json # hook wiring │ ├── agents/ # subagent definitions │ │ ├── explorer.md │ │ ├── implementer.md │ │ ├── security-reviewer.md │ │ ├── correctness-reviewer.md │ │ ├── performance-reviewer.md │ │ ├── test-reviewer.md │ │ ├── review-orchestrator.md │ │ └── mcp-operator.md │ ├── skills/ │ │ ├── pr-review/ │ │ ├── open-pr/ │ │ ├── from-jira/ │ │ ├── run-tests/ │ │ └── compact-policy/ │ └── hooks/ │ ├── block-destructive.sh │ ├── block-secrets.sh │ ├── confirm-mutations.sh │ ├── auto-format.sh │ ├── typecheck-touched.sh │ ├── record-to-mcp.sh │ └── finalize-mcp-run.sh ├── .codex/ # already exists in simple-mcp │ ├── agents/ │ └── skills/ └── scripts/ # project scripts skills call

Step one: AGENTS.md

One universal instruction file. Under 150 lines. Commands in backticks. No preamble. Only rules that push agents off their defaults.

AGENTS.md# Project: <project-name>

## Stack
- TypeScript, Node 20+, pnpm
- Vitest for tests, ESLint + Prettier

## Commands
- `pnpm dev` — run locally
- `pnpm build` — production build
- `pnpm test` — run full test suite
- `pnpm typecheck` — tsc --noEmit

## Conventions
- Use Result<T, E> over thrown exceptions
- File naming: kebab-case for files, PascalCase for components
- Imports: absolute paths via tsconfig aliases
- Commits: Conventional Commits (feat:, fix:, feat!:)

## Before opening a PR
- `pnpm typecheck` passes
- `pnpm test` passes
- Run /simplify (built-in multi-reviewer)
- Jira ticket linked in description

## MCP integration
- simple-mcp is connected. Use agent_start_run for multi-step tasks.
- Jira/GitHub/Confluence/DB credentials live in simple-mcp. Never .env.

Step two: CLAUDE.md

Delegates to AGENTS.md via @-import. Adds Claude-specific rules. Under 100 lines.

CLAUDE.md# Project instructions
See @AGENTS.md for build commands, architecture, and conventions.

## Workflow
- Start in Plan Mode. Iterate the plan. Then auto-accept edits.
- Any task with 3+ steps gets a spec before code.
- Subagents for exploration touching >5 files.
- Worktrees (claude -w) for tasks running >20 min.

## Automation
- Run /simplify before opening any PR.
- Use /batch for migrations touching >10 files.
- Use /btw for side questions — don't spawn new sessions.

## Compact policy
Preserve: specs, file paths, error messages, current simple-mcp runId.
Summarize: exploration, test output (keep failures, drop passes).
Drop: failed attempts, reverted changes, resolved clarifications.

## Corrections
When the user corrects me, I update this file or the relevant skill.
The compounding line The last rule — "when the user corrects me, I update this file" — is what turns a static setup into a compounding one. Week one you correct ten things. Month one you correct two. Month three Claude does it right the first time.

Step three: .mcp.json

Points both Claude Code and Codex at your simple-mcp:

.mcp.json{
  "mcpServers": {
    "simple-mcp": {
      "command": "/absolute/path/to/simple-mcp/run-mcp.sh",

    }
  }
}

Step four: machine-wide environment

In your ~/.zshrc or ~/.bashrc:

shell# Highest-leverage single line — cheap model for subagents
export CLAUDE_CODE_SUBAGENT_MODEL="claude-sonnet-4-6"

# Enable experimental agent teams
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

# Keys

That first line alone cuts your per-session cost by roughly 60% on typical workloads. Main orchestrator on Opus for reasoning; subagents on Sonnet for exploration, grep, test-running, file-reading.

Step five: connect simple-mcp integrations once

  1. Start simple-mcp

    In your simple-mcp repo:

    pnpm dev

    Open http://localhost:3100.

  2. Connect GitHub

    Navigate to Connections. Create a Personal Access Token at GitHub Settings → Developer settings with scopes repo and read:org. Paste it in simple-mcp. Save.

  3. Connect Jira

    Generate an API token at id.atlassian.com. Add a connection with base URL, email, token. Save.

  4. Add Confluence and databases the same way

    Credentials are AES-256 encrypted at rest. You'll never type them again.

You're done with setup Both Claude Code and Codex now know about simple-mcp. Every integration is in one encrypted vault. Subagents run on Sonnet. The rest of this handbook is building hooks, skills, and subagents on top of this foundation.
Chapter five · Build the system

Hooks — the guardrails

Deterministic rules that cost zero tokens. Hooks are where agentic-ness becomes reliable. Anything you'd repeat in every prompt goes here.

How hooks work

A hook is a shell command attached to a Claude Code lifecycle event. Events include PreToolUse (before a tool call), PostToolUse (after), Stop (session ending), SubagentStop (subagent done), UserPromptSubmit, and more.

The contract is simple:

  • Input: JSON on stdin — the tool name, arguments, output, etc.
  • Output: JSON on stdout for fine control, or an exit code (0 allow, 2 block).

For PreToolUse, you return permissionDecision: "allow" | "deny" | "ask" with an optional reason that gets shown to Claude. For PostToolUse and Stop, you use top-level decision: "block" with a reason that Claude reads as feedback.

The wiring file

.claude/settings.json connects events to scripts. Multiple hooks on the same matcher run in parallel.

.claude/settings.json{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{ "type": "command",
          "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/block-destructive.sh" }]
      },
      {
        "matcher": "Write|Edit|MultiEdit",
        "hooks": [{ "type": "command",
          "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/block-secrets.sh" }]
      },
      {
        "matcher": "mcp__simple-mcp__jira_create_issue|mcp__simple-mcp__jira_transition_issue|mcp__simple-mcp__github_review_pr",
        "hooks": [{ "type": "command",
          "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/confirm-mutations.sh" }]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit",
        "hooks": [
          { "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/auto-format.sh" },
          { "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/typecheck-touched.sh" }
        ]
      }
    ],
    "SubagentStop": [
      { "hooks": [{ "type": "command",
        "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/record-to-mcp.sh" }] }
    ],
    "Stop": [
      { "hooks": [{ "type": "command",
        "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/finalize-mcp-run.sh" }] }
    ]
  }
}

The seven hooks every project needs

1 · block-destructive.sh — the safety net

Refuses rm -rf /, force-pushes, DROP TABLE, and other irreversible operations.

bash#!/bin/bash
set -euo pipefail
INPUT=$(cat)
CMD=$(echo "$INPUT" | jq -r '.tool_input.command // ""')

BLOCKED='rm -rf /|rm -rf ~|rm -rf \*|git push --force|git reset --hard HEAD~|DROP TABLE|DROP DATABASE'

if echo "$CMD" | grep -qE "$BLOCKED"; then
  jq -n --arg cmd "$CMD" '{
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: ("Blocked: " + $cmd + ". If intentional, run it outside Claude Code.")
    }
  }'
  exit 0
fi
exit 0

2 · block-secrets.sh — prevents credential leaks

Refuses writes to .env or files containing secret patterns (API keys, private keys).

bash#!/bin/bash
set -euo pipefail
INPUT=$(cat)
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // ""')
CONTENT=$(echo "$INPUT" | jq -r '.tool_input.content // .tool_input.new_string // ""')

if [[ "$FILE" == *".env"* ]] || [[ "$FILE" == *"credentials"* ]] || [[ "$FILE" == *"id_rsa"* ]]; then
  jq -n '{
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: "Refusing to write to secret file. Use simple-mcp admin panel at localhost:3100/connections."
    }
  }'
  exit 0
fi

if echo "$CONTENT" | grep -qE '(sk-ant-[a-zA-Z0-9]{20,}|ghp_[a-zA-Z0-9]{30,}|AKIA[0-9A-Z]{16}|-----BEGIN.*PRIVATE KEY-----)'; then
  jq -n '{
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: "Detected secret pattern in content. Move to simple-mcp vault."
    }
  }'
  exit 0
fi
exit 0

3 · confirm-mutations.sh — the MCP-specific guardrail

Matches simple-mcp tools that mutate external state: jira_create_issue, jira_transition_issue, github_review_pr. Forces explicit confirmation by asking the user. This is the pattern — hooks can guard specific MCP tools by name.

bash#!/bin/bash
set -euo pipefail
INPUT=$(cat)
TOOL=$(echo "$INPUT" | jq -r '.tool_name // ""')
ARGS=$(echo "$INPUT" | jq -c '.tool_input // {}')

# Ask for user confirmation on any external mutation
jq -n --arg tool "$TOOL" --arg args "$ARGS" '{
  hookSpecificOutput: {
    hookEventName: "PreToolUse",
    permissionDecision: "ask",
    permissionDecisionReason: ("About to mutate external state via " + $tool + ". Args: " + $args)
  }
}'
exit 0

The pattern generalizes. Any simple-mcp tool that writes to an external system — creating a Jira issue, transitioning a workflow, posting a GitHub review — is a candidate for a per-tool matcher with user confirmation. Read tools (jira_search_issues, github_list_prs, github_get_pr_diff) don't need this.

4 · auto-format.sh — silent housekeeping

Runs Prettier or Ruff after every edit. Deterministic, never asks.

bash#!/bin/bash
set -euo pipefail
INPUT=$(cat)
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // ""')
[[ -z "$FILE" ]] && exit 0
[[ ! -f "$FILE" ]] && exit 0

case "$FILE" in
  *.ts|*.tsx|*.js|*.jsx|*.json|*.md)
    pnpm exec prettier --write "$FILE" >/dev/null 2>&1 || true ;;
  *.py)
    pnpm exec ruff format "$FILE" >/dev/null 2>&1 || true ;;
esac
exit 0

5 · typecheck-touched.sh — the highest-leverage hook

This one turns every edit into a feedback loop. If the type check fails, Claude sees the error on the next turn and fixes it without you saying anything.

bash#!/bin/bash
set -euo pipefail
INPUT=$(cat)
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // ""')

if [[ ! "$FILE" =~ \.(ts|tsx)$ ]]; then exit 0; fi

OUTPUT=$(pnpm exec tsc --noEmit --incremental 2>&1 || true)

if echo "$OUTPUT" | grep -qE 'error TS[0-9]+:'; then
  ERRORS=$(echo "$OUTPUT" | grep -E 'error TS[0-9]+:' | head -20)
  jq -n --arg errors "$ERRORS" '{
    decision: "block",
    reason: ("Typecheck failed. Fix these before continuing:\n" + $errors)
  }'
  exit 0
fi
exit 0

6 · record-to-mcp.sh — subagent results into simple-mcp

When a subagent finishes, its result is persisted as a step on the current simple-mcp tracked run, so the admin panel shows a full timeline across sessions.

Depends on a proposed extension As of simple-mcp v0.4, the agent_record_step capability is exposed as an MCP tool but not as a plain HTTP endpoint. Using this hook as written requires the HTTP hook-ingestion endpoint described in Chapter 16 (proposed extension P2·05). Until that ships, the same effect can be achieved by having the subagent itself call the MCP tool as part of its output. 
bash#!/bin/bash
set -euo pipefail
INPUT=$(cat)
SUBAGENT_OUTPUT=$(echo "$INPUT" | jq -r '.result // ""')
SUBAGENT_NAME=$(echo "$INPUT" | jq -r '.subagent_name // "unknown"')

RUN_FILE="/tmp/claude-mcp-run-id.txt"
[[ ! -f "$RUN_FILE" ]] && exit 0

RUN_ID=$(cat "$RUN_FILE")
SUMMARY=$(echo "$SUBAGENT_OUTPUT" | head -c 2000)

# Requires simple-mcp hook-ingestion endpoint (Chapter 16, P2·05)
curl -s -X POST "http://localhost:3101/api/agents/runs/$RUN_ID/steps" \
  -H "Content-Type: application/json" \
  -d "$(jq -n --arg name "$SUBAGENT_NAME" --arg summary "$SUMMARY" \
    '{title: ("subagent: " + $name), summary: $summary, kind: "subagent_result"}')" \
  >/dev/null 2>&1 || true
exit 0

7 · finalize-mcp-run.sh — clean close on session end

bash#!/bin/bash
set -euo pipefail
RUN_FILE="/tmp/claude-mcp-run-id.txt"
[[ ! -f "$RUN_FILE" ]] && exit 0

RUN_ID=$(cat "$RUN_FILE")
curl -s -X POST "http://localhost:3101/api/agents/runs/$RUN_ID/complete" \
  -H "Content-Type: application/json" \
  -d '{"status": "completed"}' >/dev/null 2>&1 || true

rm -f "$RUN_FILE"
exit 0
Do not forget chmod +x .claude/hooks/*.sh — once, after creating these. Without it, hooks silently fail and you'll spend an hour wondering why nothing's firing. This is the single most common gotcha in all of agentic setup.
Chapter six · Build the system

Skills — the recipes

A skill is a directory Claude loads on demand. Write once, invoke forever. The format works across Claude Code, Codex, Cursor, and every other compatible agent.

Anatomy of a skill

A skill is a directory, not a file. The entry is SKILL.md with YAML frontmatter. Claude reads the description field to decide whether to load the skill. Supporting files — scripts, references, examples — are loaded only when the skill specifically needs them. This is progressive disclosure: a 500-line skill costs zero tokens until triggered.

.claude/skills/pr-review/ ├── SKILL.md # required — frontmatter + trigger logic ├── scripts/ │ └── merge-reviews.ts # deterministic merge logic ├── references/ │ └── severity-rules.md # loaded only when Claude needs detail └── examples/ └── good-report.md # shows the expected output

The four rules for writing skills

  1. Description is a trigger, not a summary

    Write "Use when the user provides a Jira key or Jira URL" — not "This skill fetches Jira tickets". The description is what Claude reads to decide whether this skill applies.

  2. Don't state what Claude already does

    A skill that says "be helpful and accurate" is wasted tokens. Focus only on what pushes Claude off its default behavior.

  3. Give goals and constraints, not step-by-step instructions

    Except when the steps are the whole point (like the merge pipeline). Over-prescriptive skills make Claude brittle.

  4. Include scripts so Claude composes instead of reconstructing

    A skill that says "run ./scripts/parse-vitest.sh" is better than a skill that re-derives parser logic every time.

Skill 1 · from-jira — turn a ticket into a spec

This is the highest-leverage workflow skill. Give it a Jira key, get back a ready-to-implement spec.

.claude/skills/from-jira/SKILL.md---
name: from-jira
description: Converts a Jira ticket into an implementation-ready spec. Use when the user provides a Jira issue key (like PROJ-1234) or a Jira URL. Fetches the ticket, linked Confluence pages, and related code, then produces acceptance criteria.
---

# Jira ticket → spec

## Steps
1. Extract the Jira key from the user's message (pattern: [A-Z]+-\d+)
2. Call mcp__simple-mcp__jira_search_issues with JQL `key = <KEY>`
3. Scan the description for Confluence URLs; fetch each via
   mcp__simple-mcp__confluence_get_page
4. Scan for referenced file paths; search for each via
   mcp__simple-mcp__fs_workspace_search
5. Produce the spec using references/spec-template.md

## Output shape
- Context: 2–3 sentences on why this ticket exists
- Goal: one sentence
- Acceptance criteria: bullet list, each testable
- Files to touch: paths with one-line reason each
- Out of scope: what the ticket does NOT include
- Open questions: things the user must clarify

## Stop conditions
- Empty acceptance criteria + short description → ask user
- Multiple conflicting tickets → ask user
- Never start implementing. Spec only.

## Gotchas
- Jira descriptions often have truncated code references. Always search the
  workspace for confirmations.
- Confluence pages may be gated; if a fetch 403s, note it and proceed.

Skill 3 · open-pr — the PR-creation recipe

Encodes your team's PR process so Claude doesn't reinvent it every time.

.claude/skills/open-pr/SKILL.md---
name: open-pr
description: Opens a PR with conventional-commit branch name, generated description, and linked Jira ticket. Use only after pr-review has run and user has approved. Runs pnpm typecheck and pnpm test as a final gate.
---

# Open a PR

## Preconditions — refuse if not met
- pr-review or /simplify has run this session
- User has explicitly said "open the PR" or "create the PR"
- `git status` is clean (no uncommitted changes)

## Steps
1. `pnpm typecheck && pnpm test` — if either fails, stop and report
2. Get commit messages since main; extract Jira key if present
3. Build PR description: summary, bullet list of changes, test plan, Jira link
4. `gh pr create --fill-verbose --body-file /tmp/pr-body.md`
5. Return the PR URL

## Never
- Push with --force
- Bypass branch protection
- Open a PR with unaddressed pr-review findings

Skill 4 · run-tests — focused test output

.claude/skills/run-tests/SKILL.md---
name: run-tests
description: Runs the project's test suite and returns only failures with file:line. Use after any code change. Never dumps full test output.
---

# Run tests

Subagents — the team

Specialists with their own context windows, tool whitelists, and system prompts. Use them to parallelize work and to keep your main context clean.

The four rules for subagents

  • One clear goal per agent. If you need two things, write two agents.
  • Whitelist tools explicitly. Omitting tools: inherits everything, including all MCP tools. This is almost never what you want.
  • Pick the model per agent. Exploration on Sonnet. Deep review on Opus. Implementation on Sonnet. Default to Sonnet; promote to Opus only where judgment matters.
  • Describe when to fire, not what the agent is. The description is a trigger for auto-invocation.

The merge script

Deterministic merge — no LLM cost.

typescript#!/usr/bin/env tsx
import { readFileSync } from "fs";

type Finding = {
  severity: "blocker" | "should-fix" | "nice-to-have";
  file: string; line: number; message: string; suggestion?: string;
};
type Review = { reviewer: string; findings: Finding[]; summary: string };

const reviews: Review[] = process.argv.slice(2)
  .map(f => JSON.parse(readFileSync(f, "utf8")));

const all = reviews.flatMap(r => r.findings.map(f => ({ ...f, reviewer: r.reviewer })));

// Dedupe: same file+line+severity from different reviewers → merge
const key = (f: any) => `${f.file}:${f.line}:${f.severity}`;
const grouped = new Map<string, any[]>();
for (const f of all) {
  const existing = grouped.get(f.file) ?? [];
  const dup = existing.find(e => key(e) === key(f));
  if (dup) {
    dup.reviewers.push(f.reviewer);
    dup.message = `${dup.message} / ${f.message}`;
  } else {
    existing.push({ ...f, reviewers: [f.reviewer] });
    grouped.set(f.file, existing);
  }
}

console.log("# PR review\n");
for (const sev of ["blocker", "should-fix", "nice-to-have"] as const) {
  const matches = [...grouped.values()].flat().filter(f => f.severity === sev);
  if (!matches.length) continue;
  console.log(`## ${sev.toUpperCase()}`);
  for (const f of matches) {
    console.log(`- **${f.file}:${f.line}** [${f.reviewers.join(",")}] ${f.message}`);
    if (f.suggestion) console.log(`  fix: ${f.suggestion}`);
  }
  console.log();
}
console.log("## Per-reviewer summaries");
for (const r of reviews) console.log(`**${r.reviewer}**: ${r.summary}`);
Why this beats /simplify alone /simplify is Anthropic's built-in 3-reviewer pipeline (routine issues, no configuration). The custom 4-reviewer pipeline described here is for sensitive domains — auth, payments, PII, data mutations — where specialists with Opus-level judgment and structured JSON output (parseable by CI) are worth the extra cost. Use /simplify as the default; use the custom pipeline on demand.

Chapter thirteen · Operate

Token & time optimization

Seven levers, ordered by impact. The first three cover most of the gain. The rest compound over weeks.

  1. Subagent model routing — ~60% cost reduction

    CLAUDE_CODE_SUBAGENT_MODEL=claude-sonnet-4-6

    Main orchestrator on Opus. Subagents on Sonnet. Exploration, grep, file reading, tests — all on Sonnet at ~5× lower cost. Opus only where judgment actually matters.

  2. Context isolation via subagents — ~40% input token reduction

    A main session that reads 50 files has 50 files in its context forever. A main session that delegates exploration gets a 300-token summary; the 50 files never enter its context. This compounds on every subsequent turn.

  3. Parallel subagents — 4× wall-clock speedup

    Four reviewers in parallel = 4× faster than sequential, same token count. You pay for tokens either way, but iteration speed compounds.

  4. Skills loaded on demand

    A 500-line SKILL.md costs zero until triggered. Thin CLAUDE.md (under 100 lines) is always in context; skills load only when their description matches.

  5. Hooks instead of prompts

    Format-on-save, typecheck-on-edit, block-write-SQL — zero tokens. Every recurring instruction moved from CLAUDE.md to a hook saves tokens on every turn for the rest of the session.

  6. Client-driven simple-mcp runs

    Use agent_start_run (tracked only) not agent_execute (runs its own Anthropic loop). One Anthropic bill instead of two for the same work.

  7. Compact policy

    In CLAUDE.md, specify what /compact should preserve (specs, errors, file paths) and what to drop (exploration, failed attempts). Policy-driven summarization beats generic summarization.

Measured numbers

On a typical PR review in a large codebase:

Chapter fourteen · Operate

Observability & governance

You can't tune what you don't measure. Your simple-mcp admin panel is your control room.

The four metrics to track weekly

  1. Tokens per PR

    Total input + output across all runs tied to a branch. Trending up = too much exploration in main context. Fix: more subagents.

  2. Wall-clock per PR

    First commit to merge. Not going down = not parallelizing enough. Fix: /batch, worktrees, parallel subagents.

  3. Reviewer find rate

    How many blockers did each reviewer catch? Low-rate reviewers get pruned or re-tuned.

  4. Hook fire rate

    How many typecheck-touched blocks per session? Trending up means Claude is typing sloppier — tune CLAUDE.md.

A recurring review ritual

15 minutes, weekly. Open localhost:3100/agent-executions and look for:

  • Runs over 100k tokens — investigate why
  • Subagents that ran >10 times with no findings — delete or narrow them
  • Skills that never auto-triggered — description is wrong, rewrite it
  • Hooks that never fired — matcher is wrong, or not needed

Governance — hard limits

Recommended defaults for any agent_start_run from Claude Code:

call args{
  "config": {
    "maxIterations": 10,
    "maxToolCalls": 30,
    "maxTokens": 50000
  }
}

simple-mcp's built-in defaults (25 / 100 / 200k) are appropriate for server-side agent_execute runs, where the engine benefits from headroom to plan and reflect. Tighter limits for client-driven runs catch runaway loops faster — Claude Code has its own reasoning budget; simple-mcp just tracks the execution.

The admin panel as unified dashboard

Pages worth opening regularly:

  • /agent-executions — every tracked run. Iterations, tool calls, tokens, duration. Catches runaway loops.
  • /databases — token consumption log. Review weekly; anything >100k tokens warrants investigation.
  • Filter by agent — which pre-built agents are actually being used? Deprecate the dead ones.

Jira: jira_search_issues, jira_create_issue, jira_transition_issue. GitHub: github_list_prs, github_get_my_prs, github_review_pr, github_get_pr_diff, github_search_code. Agents: agent_execute, agent_status, agent_list. System: health_check, list_connections. Accessible from any MCP-compatible client over stdio or SSE.

✓ 02

Seven pre-built autonomous agents

jira-triage, pr-review, code-search, sprint-planning, local-repo-analysis, confluence-reader, database-explorer. Each agent has declared dependencies (integrations it needs), a system prompt, and a curated tool list. Dependency readiness is exposed through agent_list and visible in the admin panel.

✓ 03

Dual-mode execution

Server-side autonomous via agent_execute: simple-mcp reasons with Anthropic, plans tasks, invokes tools, reflects, self-corrects until completion or a guardrail fires. Client-driven tracked runs via agent_start_run / agent_record_step / agent_update_task / agent_complete_run: Claude Code or Codex owns the reasoning, simple-mcp persists execution history without charging a second reasoning loop.

✓ 04

Guardrails on every execution

Max iterations (default 25), max tool calls (default 100), max tokens (default 200,000), 5-minute timeout, cycle detection on 3 identical calls, delegation depth cap of 2. All configurable per-run via the config parameter.

✓ 05

React 19 admin panel

Connection management, PR dashboard, review history, agent configuration, and the execution monitoring UI with auto-refreshing detail pages, cancel buttons for active runs, and full metrics (iterations, tool calls, input/output tokens, duration). TanStack Router + TanStack Query; file-based routing; type-safe API client.

✓ 06

Encrypted credential vault

AES-256-CBC encryption for all stored credentials (Jira email + API token, GitHub PAT). One vault serves both Claude Code and Codex — the same integrations are accessible from either client without re-configuration.

✓ 07

Both .claude/ and .codex/ configurations

The repo ships with a universal AGENTS.md and CLAUDE.md. Three Codex roles live in .codex/agents/: business-analyst for requirements, senior-fullstack-typescript-developer for implementation, typescript-pr-reviewer for review. Matching skills under .codex/skills/. This is a working reference implementation of the shared-instruction pattern described in Chapter 4.

✓ 08

HTTP API for external callers

Hono server on port 3101 exposes POST /api/agents/execute, GET /api/agents/runs, GET /api/agents/runs/:runId, POST /api/agents/runs/:runId/cancel. Useful for scripting, CI integration, and the automation tiers described in Chapter 10.

✓ 09

Inter-agent delegation

Agents can delegate sub-tasks to other agents via an internal delegate_to_agent tool. Max depth 2. Parent context is summarised to 2000 characters when passed to the child. Both parent and child runs are persisted and visible in the admin panel — full lineage.

✓ 10

Release pipeline

release-please + conventional commits + automated CHANGELOG. PR merges to main trigger CI (pnpm typecheck, pnpm test, pnpm build). A release PR is opened and updated by the workflow; merging it bumps the version, tags, and publishes a GitHub Release.

Chapter sixteen · simple-mcp reference

Proposed extensions

Eight proposed extensions to simple-mcp. They address specific scenarios that become interesting once the workflows in this handbook are adopted at team or production scale. Contributions welcome — see the repo's issues tab.

Scope These are design proposals, not committed roadmap items. Priority labels (P1/P2/P3) reflect impact on the workflows in this handbook, not project priorities. The existing pr-review and six sibling agents are fully functional today — these extensions are about additional capabilities.

P1 — Highest impact on handbook workflows

P1 · 01

Register additional pre-built agents

The seven existing agents cover common workflows. Two additions would complete the handbook's pipeline coverage:

  description: "Scans dependencies and recent changes for CVEs + secrets",
  requiredIntegrations: ["github"],

  requiredTools: ["github_search_code", "github_list_prs"]

P1 · 02

Remote-accessible deployment mode

As of v0.4, simple-mcp binds to localhost. Claude Code routines and any other cloud-hosted agent can't reach it. Three viable approaches:

  • Option A (easiest): Bearer-token HTTP endpoint. Add CLAUDE_MCP_PUBLIC_URL and CLAUDE_MCP_API_TOKEN env vars. Deploy to Fly.io, Render, or similar. Cloud callers POST to $PUBLIC_URL/api/agents/execute with Authorization: Bearer $TOKEN.
  • Option B (middleweight): HTTPS MCP transport. Add CLAUDE_MCP_TRANSPORT=http support with bearer auth. Cloud callers connect as a proper remote MCP server.
  • Option C (workaround): GitHub-webhook bridge. Cloud routine opens a PR on a claude/ branch. Local simple-mcp instance picks it up via /loop and writes locally.

Recommendation: start with A. C is a decent backup that avoids public exposure.

P1 · 03

Per-run budget enforcement on client-driven runs

Server-side agent_execute already enforces guardrails (iterations / tool calls / tokens). Client-driven runs (agent_start_run + agent_record_step) currently rely on the caller to respect limits. Extension: store guardrails on the run at start time; reject steps that would breach them.

typescript// on agent_record_step
if (run.inputTokensUsed + delta > run.guardrails.maxTokens) {
  await updateRunStatus(runId, "cancelled", "budget_exceeded");
  throw new Error("Run exceeded token budget");
}

Protects against buggy client sessions that keep recording steps past their configured budget.

P2 — Team-scale observability

P2 · 04

Skill & subagent registry in the admin panel

The admin panel tracks executions but not the definitions (skills, subagents) that produced them. Add tables + UI for:

  • Skills discovered in .claude/skills/ of connected repos — with description, last-triggered timestamp, and invocation count
  • Subagents discovered in .claude/agents/ — same metadata

Makes the admin panel the source of truth for "what capabilities does this workspace have?" and surfaces dead skills/agents during review.

P2 · 05

HTTP hook-ingestion endpoint

Claude Code hooks (described in Chapter 5) can't call MCP tools directly — they're shell scripts. They need an HTTP target:

endpointsPOST /api/hooks/event
  body: { eventName, tool, toolInput, toolOutput, runId?, sessionId }
  → stores to hook_events table, correlates to run if runId present

POST /api/agents/runs/:runId/steps
  body: { title, summary, kind }
  → thin wrapper that calls agent_record_step internally

Enables the record-to-mcp.sh and finalize-mcp-run.sh hooks described in Chapter 5 to work as written, and gives the admin panel a full session timeline (every hook fire alongside every tool call and subagent).

P2 · 06

Cost tracking with per-model breakdown

Input/output tokens are already tracked per run. Extensions:

  • model column on each step (e.g., claude-sonnet-4-6, claude-opus-4-7)
  • Computed USD cost based on a pricing table
  • Daily / weekly / monthly rollups in the admin panel
  • Optional budget alerts via Slack webhook on 80% / 100% of a configured monthly cap

Turns the admin panel from "execution dashboard" into "cost control dashboard."

P3 — Polish & governance

P3 · 07

Enforced MCP tool permission matrix per agent

Agents currently declare requiredTools. Make it authoritative: at agent_start_run, store the run's tool whitelist. Reject agent_record_step for tools outside the whitelist. Prevents scope drift when a caller accidentally uses a tool the agent shouldn't need.

P3 · 08

Run replay and diff

For any completed run, expose "replay with same config" — useful for debugging agent logic after a prompt change. Plus "diff between runs" for comparing agent versions side-by-side. Both are backend-cheap (the data is already persisted) but admin-panel changes.

P3 · 09

Webhook fan-out on run completion

When a run completes, fire a configured webhook with a structured payload. Subscribers (Slack, Linear, CI, external dashboards) get notified without polling. This is how simple-mcp grows from "Claude Code companion" into "node in a larger automation graph."

What these extensions add up to

simple-mcp becomes the observability, governance, and orchestration layer for all agentic work on a machine or team — Claude Code, Codex, and cloud routines — with one credential vault, one run history, one cost dashboard. — the north star for simple-mcp v1

None of these require rewriting existing code. They are additive — each module sits next to what's already there. P1s can be targeted in a single iteration; P2s and P3s over subsequent minor versions.

Appendix B

Pitfalls & gotchas

The mistakes everyone makes on their first week. Save yourself the debugging time.

Setup gotchas

!

Hook scripts not executable

chmod +x .claude/hooks/*.sh — once, after creating hooks. Without it, hooks silently fail and you'll spend an hour wondering why nothing's firing. The single most common gotcha in all of agentic setup.

!

Hook stdout polluted

Don't echo debug info in hook scripts — it corrupts the JSON parser. Log to a file, or use stderr. Exit 0 only when your hook is 100% silent or outputs valid JSON.

!

Shell profile printing on startup

If your .zshrc prints a welcome banner, that text will appear on hook stdout when scripts source your profile. Break the JSON parse, break the hook. Move banners behind an [[ -o interactive ]] guard.

Subagent gotchas

!

Subagent with inherited tools

Omitting tools: in frontmatter inherits everything, including all MCP tools. Subagents then wander. Always whitelist.

!

Sequential dispatches when you wanted parallel

If you don't explicitly request parallel, Claude often sequences subagent calls. Say "dispatch all four in parallel" in the orchestrator prompt. The Task tool accepts multiple invocations in a single call — that's how you get true parallelism.

Skill gotchas

!

Description is a summary

"This skill opens pull requests" — Claude won't load it when you ask to open a PR. Write "Use when the user asks to open a pull request". Triggers, not summaries.

!

Skill folder without SKILL.md

A directory with just scripts or templates will be silently ignored. Every skill needs SKILL.md at the root of its directory.

simple-mcp gotchas

!

Running agent_execute when you meant agent_start_run

Double-bill. agent_execute runs its own Anthropic loop server-side. Use agent_start_run from within Claude Code to keep the reasoning in your existing session.

!

Stale MCP process after code changes

MCP loads code at process start. After changing simple-mcp code, restart: in Claude Code type /mcp, select simple-mcp, restart. Or pkill -f 'tsx.*cli.ts start'.

Workflow gotchas

!

Too much in CLAUDE.md

Keep CLAUDE.md under 100 lines. Anything reusable → skill. Anything deterministic → hook. CLAUDE.md is advisory (followed ~70% of the time); overstuffing it dilutes the important rules.

!

Skipping Plan Mode on complex tasks

Boris's rule: tasks with 3+ steps get a spec first. Skipping Plan Mode is how you end up with 40 changes you didn't want. Plan Mode first, auto-accept second.
